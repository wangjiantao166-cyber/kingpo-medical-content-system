# Product Maintenance Workflow

## Goal

This workflow lets KingPo add, update, review, and retire product pages without losing source control, technical accuracy, SEO quality, or medical/regulatory caution.

## Workflow Stages

### 1. Source Intake

Collect available materials:

- Product spreadsheet
- PDF catalog
- Product photos
- Existing official website page
- Internal technical note
- Boss/user confirmation
- Approved diagrams
- Standard mapping notes

Classify each source:

- Owned official source
- User-confirmed source
- Historical source
- Internal reference
- Competitor reference
- Needs verification

### 2. Product Record Creation

Create a structured product record with:

- Name
- Model
- Category
- Status
- Test object
- Test item
- Standard direction
- Technical parameters
- Assets
- Source notes
- Verification state

### 3. Status Decision

Assign one product status:

- Existing product
- Custom development available
- R&D direction
- R&D reference / needs confirmation
- Internal only
- Not public
- Legacy product
- Discontinued

### 4. Standards Review

Before writing a public page:

- Map standards carefully
- Classify relationship type
- Confirm version or mark as unknown
- Avoid full-compliance claims unless verified
- Record evidence source

### 5. Draft Generation

Generate a WordPress draft using the approved product page structure:

1. H1
2. One-sentence positioning
3. Core information
4. Product overview
5. Applicable standards and clause direction
6. Test principle
7. Test object and sample type
8. Test items
9. Technical parameters
10. Typical applications
11. R&D validation / registration testing / third-party lab scenarios
12. Selection and configuration notes
13. Related products
14. FAQ
15. Inquiry CTA

### 6. Asset Review

Check:

- Image ownership
- PDF public permission
- Alt text
- File names
- Version/date
- Duplicate assets

### 7. SEO and Internal Links

Prepare:

- SEO title
- Meta description
- URL slug
- Core keywords
- Related keywords
- FAQ
- Schema recommendation
- Internal links

### 8. Publish QA

A page can be published only when:

- Product status is correct
- Sources are recorded
- Standards are not overstated
- Parameters have evidence or are marked pending
- Assets are approved
- Inquiry CTA is present
- Internal links are complete
- Mobile layout is checked

### 9. Update and Refresh

Every product should have:

- Last review date
- Next review date
- Change log
- Owner/reviewer

Refresh triggers:

- New PDF version
- New product parameter
- New standard version
- Broken link
- Old image
- Inquiry quality issue
- SEO underperformance

### 10. Retirement

For discontinued or legacy products:

- Decide whether to keep, noindex, merge, or redirect
- Preserve source records
- Do not delete pages without a URL plan
